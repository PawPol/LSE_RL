### RNG stream design choice (documented in code)

`act()` draws a uniform candidate action *unconditionally* every step,
then either returns `last_action` (with probability `inertia_lambda`)
or the candidate (with probability `1 - inertia_lambda`). The
alternative — draw the candidate only when *not* sticking — yields an
identical marginal stationary distribution but a different per-seed
realised stream.

The chosen variant has two operational benefits:
1. Per-step RNG consumption is constant (one `integers` + one `random`
   draw on every call after the first), so the stream is robust to
   callers that branch on intermediate state.
2. Matches the M3 dispatch's calibrated smoke threshold
   (`counts.max() > 30` after 50 steps at `inertia_lambda = 0.9`,
   `seed = 0`), which is satisfied with the candidate-first variant
   but only ~50% of seeds for the "switch-only" variant — the
   threshold is statistically loose at 50 steps with `λ = 0.9`
   (expected switch count is 5; concentration on a single action
   depends on luck).

This is documented inline in `act()`'s docstring under "Implementation
note (RNG stream order)".

### Auxiliary sanity checks (all PASS)

Run via `PYTHONPATH=. .venv/bin/python -c "..."`:

1. **`inertia_lambda = 0` (degenerate uniform).** Over 2000 steps,
   action counts are `[514, 499, 500, 487]` — uniform within Monte
   Carlo noise.
2. **`inertia_lambda = 1` (degenerate lock-in).** After the first
   uniform draw, all subsequent 99 actions equal the first. Lock-in
   verified.
3. **Boundary / NaN rejection.** `ValueError` raised for each of
   `-0.1`, `1.1`, `NaN`, `+inf` with the expected message
   `"inertia_lambda must lie in [0, 1], got <value>"`.
4. **`n_actions = 1` edge case.** All 10 actions equal `0`; no
   off-by-one in the `integers(0, n_actions)` draw.
5. **`reset(seed=42)` reproducibility.** Two consecutive 20-step
   action streams with `reset(seed=42)` between them are identical;
   confirms `last_action` is cleared and the RNG is re-seeded.
6. **`info()` mandatory key coverage.** All keys in
   `ADVERSARY_INFO_KEYS` are present (validated by `_validate_info`
   in `base.py`); `phase = "inertia"`, `inertia_lambda = 0.5`,
   `last_action = None` before any `act()`, then populated to the

## Open questions / for downstream agents

1. **Test-author (M3 W1.B).** Spec §13.3 lists
   `tests/adaptive_beta/strategic_games/test_inertia_adversary.py` as
   the M3 deliverable for this adversary. Suggested invariants to
   cover (in addition to the smoke checks above):
   - `inertia_lambda` validation: rejects `<0`, `>1`, NaN, `±inf`.
   - Boundary `λ = 0` reduces to a uniform adversary (statistical
     test: KS / chi-squared on action counts over a long horizon).
   - Boundary `λ = 1` locks onto the first action forever
     (deterministic, no Monte Carlo).
   - `info()` returns a dict that is a superset of
     `ADVERSARY_INFO_KEYS` and that `phase == "inertia"`,
     `inertia_lambda` matches the constructor argument,
     `last_action` is `None` before any `act()` and equals the last
     returned action afterwards.
   - Determinism under fixed `(n_actions, inertia_lambda, seed)`
     (compare two independently-constructed adversaries' first 100
     actions).
   - `reset(seed=...)` re-seeds AND clears `last_action`.
   - `n_actions = 1` always returns `0`.
   - Empirical stickiness rate (fraction of `t > 0` steps where
     `a_t == a_{t-1}`) approaches `inertia_lambda + (1 - inertia_lambda) / n_actions`
     as `T → ∞` for `λ ∈ {0.0, 0.3, 0.7, 0.9}` (this is the
     candidate-first variant's exact stationary stickiness;
     statistical tolerance ≥ 3σ over `T = 10000`).
